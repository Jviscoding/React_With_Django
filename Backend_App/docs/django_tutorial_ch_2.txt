* CREATING models
        -> A model is the single, definitive source of information about your data. 
        -> It contains the essential fields and behaviors of the data you’re storing.
        -> in short term, it is python's way of creating sqql command via ORM rather than using sql syntax

        -> when calling migration, this models will be modified and executes calls on database which modifies database when needed

        -> for creating a models, use this:

            from django.db import models


            class Question(models.Model):
                question_text = models.CharField(max_length=200)
                pub_date = models.DateTimeField("date published")


            class Choice(models.Model):
                question = models.ForeignKey(Question, on_delete=models.CASCADE)
                choice_text = models.CharField(max_length=200)
                votes = models.IntegerField(default=0)

    
    * KEEP IN MIND WITH MODELS
        * Django models are Python classes that inherit from `django.db.models.Model`.

        * Class variables inside a model represent database fields (columns).

        * Fields use Django Field classes (e.g., `CharField`, `DateTimeField`) to define the type of data stored.

        * The variable name of a field becomes:
                * The field name used in Python code.
                * The column name in the database.

        * An optional first argument can provide a human-readable field name; otherwise Django uses the variable name.

        * Some fields require arguments (e.g., `CharField(max_length=...)`).

        * Fields can also have optional arguments (e.g., `default=0`).

        * Relationships are defined using fields such as `ForeignKey`.

        * `ForeignKey` creates a many-to-one relationship (e.g., many Choices belong to one Question).
        * at the sample one, Question's id(key) will be used as reference for the question column of the Question table's id

        * Django supports common relationships:
            * Many-to-One (`ForeignKey`)
            * Many-to-Many (`ManyToManyField`)
            * One-to-One (`OneToOneField`)

* SQL Migrate
    (terminal) python manage.py sqlmigrate polls 0001
        -> if you want to see the structure in SQL format and this will show something like this:
        BEGIN;
            --
            -- Create model Question
            --
            CREATE TABLE "polls_question" (
                "id" bigint NOT NULL PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
                "question_text" varchar(200) NOT NULL,
                "pub_date" timestamp with time zone NOT NULL
            );
            --
            -- Create model Choice
            --
            CREATE TABLE "polls_choice" (
                "id" bigint NOT NULL PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
                "choice_text" varchar(200) NOT NULL,
                "votes" integer NOT NULL,
                "question_id" bigint NOT NULL
            );
            ALTER TABLE "polls_choice"
            ADD CONSTRAINT "polls_choice_question_id_c5b4b260_fk_polls_question_id"
                FOREIGN KEY ("question_id")
                REFERENCES "polls_question" ("id")
                DEFERRABLE INITIALLY DEFERRED;
            CREATE INDEX "polls_choice_question_id_c5b4b260" ON "polls_choice" ("question_id");

        COMMIT;

        * KEEP IN MIND 

            - SQL output depends on DB backend (PostgreSQL, MySQL, SQLite, etc.).
            - Table names auto-generated as: <app>_<model_lowercase>
            e.g. polls_question, polls_choice (can be overridden).

            - Primary key (id) is auto-created unless explicitly defined.

            - ForeignKey fields:
            - Stored in DB as <field_name>_id
            - Can be customized if needed.

            - Foreign key relationships:
            - Enforced via FOREIGN KEY constraint
            - DEFERRABLE is PostgreSQL-specific (delays constraint checking until transaction end)

            - Django abstracts DB differences:
            - Auto-increment / identity fields handled automatically per DB engine
            - Quoting rules handled internally (PostgreSQL, MySQL, SQLite differences)

            - sqlmigrate command:
            - DOES NOT execute migrations
            - Only prints SQL Django will run
            - Used for review or DB admin approval

            - check command:
                python manage.py check

            - Validates project for issues
            - Does not touch database or run migrations

Django Shell & ORM (Short Notes)
    `
    - Start shell with:
    python manage.py shell
    (uses DJANGO_SETTINGS_MODULE and auto-loads INSTALLED_APPS models)

------------------------------------------------------------
BASIC ORM OPERATIONS
------------------------------------------------------------

    - View all objects:
    Question.objects.all()
    → returns QuerySet

    - Create object:
    from django.utils import timezone
    q = Question(question_text="What's new?", pub_date=timezone.now())
    q.save()

    - Access fields:
    q.id
    q.question_text
    q.pub_date

    - Update object:
    q.question_text = "What's up?"
    q.save()

------------------------------------------------------------
STRING REPRESENTATION
    ------------------------------------------------------------

    - Default output is not readable:
    <Question: Question object (1)>

    - Fix using __str__():
    class Question(models.Model):
        def __str__(self):
            return self.question_text

    class Choice(models.Model):
        def __str__(self):
            return self.choice_text

    - Used in shell, admin panel, and debugging

------------------------------------------------------------
CUSTOM METHODS
------------------------------------------------------------

    - Example method:
    def was_published_recently(self):
        return self.pub_date >= timezone.now() - timedelta(days=1)

    - Requires:
    import datetime
    from django.utils import timezone

------------------------------------------------------------
QUERYING DATA
------------------------------------------------------------

    - Filter:
    Question.objects.filter(id=1)
    Question.objects.filter(question_text__startswith="What")

    - Get single object:
    Question.objects.get(pub_date__year=current_year)

    - Primary key shortcut:
    Question.objects.get(pk=1)

    - Non-existent object:
    → raises DoesNotExist exception

------------------------------------------------------------
RELATIONSHIPS (FOREIGN KEY)
------------------------------------------------------------

    - Access related objects:
    q.choice_set.all()
    q.choice_set.count()

    - Create related object:
    q.choice_set.create(choice_text="Not much", votes=0)

    - Reverse relation:
    c.question  → parent Question

    - Filter across relationships:
    Choice.objects.filter(question__pub_date__year=current_year)

------------------------------------------------------------
DELETION
------------------------------------------------------------

    - Delete object:
    c.delete()

------------------------------------------------------------
KEY IDEA
------------------------------------------------------------

    - ORM lets you work with DB using Python objects
    - ForeignKey creates automatic reverse manager: <model>_set
    - Double underscore (__) is used for field traversal and filtering`

------------------------------------------------------------------------------------------------------------------
DJANGO LOOKUP

Django ORM Lookup: pub_date__year

    - pub_date__year is NOT a variable
    - It is NOT a model field
    - It is NOT a database column

------------------------------------------------------------
WHAT IT IS
------------------------------------------------------------

    - It is a Django ORM "field lookup expression"
    - Used to filter or query based on parts of a field

------------------------------------------------------------
HOW IT WORKS
------------------------------------------------------------

    Given a model field:
    pub_date = DateTimeField()

    Django allows you to access parts of it using "__" (double underscore):

    pub_date__year

    Meaning:
    → Extract the YEAR from pub_date
    → Compare it in the query

------------------------------------------------------------
EXAMPLE
------------------------------------------------------------

    Question.objects.get(pub_date__year=current_year)

    Breakdown:
    - pub_date → datetime field
    - __year → extract year part
    - current_year → Python variable (e.g. 2026)

------------------------------------------------------------
SQL EQUIVALENT (approximate)
------------------------------------------------------------

    SELECT *
    FROM question
    WHERE EXTRACT(YEAR FROM pub_date) = 2026;

------------------------------------------------------------
KEY IDEA
------------------------------------------------------------

    - Django uses "__" to "drill into" field data
    - Enables filtering on:
    - dates (year, month, day)
    - strings (startswith, contains)
    - relationships (foreign keys)
